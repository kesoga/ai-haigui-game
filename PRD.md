# AI 海龟汤游戏 PRD（更新版）

版本：v1.1（已集成三值合规、重试/兜底、一致性与埋点等关键改动）  
作者：产品团队  
更新时间：2026-03-21

---

## 1. 产品概述
AI 驱动的海龟汤对话推理游戏。玩家与 AI 主持人通过提问获取线索并推理故事真相。AI 主持人对玩家的每个问题只返回三值之一：`是` / `否` / `无关`。系统在后端强制该输出格式，并包含重试与兜底策略以保障稳定性与公平性。

---

## 2. 目标用户
- 喜欢推理游戏的玩家  
- 想要轻松娱乐的上班族  
- 对 AI 应用感兴趣的技术爱好者

---

## 3. 核心目标
1. 提供快速开局、低门槛的海龟汤玩法。  
2. 将 AI 主持人能力产品化：稳定地产出三值线索反馈。  
3. 提供“查看汤底”的仪式感，形成完整闭环（提问 -> 揭晓 -> 再来一局）。

---

## 4. 成功指标（上线后可量化）
- 完局率：进入游戏页面后，点击“查看汤底/结束游戏”的比例  
- 会话完成效率：查看汤底前的平均提问次数（期待合理区间波动）  
- 留存指标：D1、D7留存 / “再来一局” 点击率  
- 体验质量：AI 响应失败率、平均响应时间、三值合规率（目标 >= 99%）、重试/兜底率

---

## 5. 用户旅程（端到端）
1. 游戏大厅：浏览游戏列表 -> 选择难度/类型 -> 点击“开始游戏”。  
2. 游戏页面：看到汤面故事 -> 在聊天输入框提问 -> AI 返回三值反馈 -> 可随时查看对话历史。  
3. 汤底页面：查看完整汤底与玩家推理过程 -> 点击“再来一局”回到大厅或直接开启新局。  
- 当玩家接近单局上限时（配置阈值，如剩余 <= 5 问），UI 展示提醒。

## 6. 功能需求

### 6.1 MVP（必须做）
包含原要点并新增后端合规与埋点约定。

#### 6.1.1 游戏大厅
- 展示游戏列表（难度：简单/中等/困难；类型：时间线/人物/事件等）。  
- 卡片式布局：故事标题/简述、难度标签、类型标签。  
- 支持点击“开始游戏”进入对应游戏页面。

#### 6.1.2 游戏页面
- 展示汤面故事（玩家可见线索）。  
- 聊天输入框：玩家输入问题，每次提问触发 AI 主持人回应。  
- AI 输出约束（必须严格遵守）：
  - 仅返回三种之一：`是` / `否` / `无关`。  
  - 任何包含解释、额外文本、标点或其它词汇的输出视为不合格，后端触发重试/兜底策略（默认重试 N 次，最终返回 `无关` 并记录原始响应）。  
- 显示对话历史（按时间顺序，玩家问题与 AI 三值反馈）。  
- 按钮：`查看汤底`（进入汤底页面并展示完整汤底）、`结束游戏`（结束会话并由产品决定跳转：大厅或汤底页）。

#### 6.1.3 汤底页面
- 展示完整汤底（故事真相）。  
- 展示玩家推理过程（基于对话历史整理“问题 -> AI 反馈”的推理轨迹）。  
- `再来一局` 按钮：进入新局并清空对话历史与推理过程。  
- 揭晓可具仪式感（淡入/逐步展开/倒计时等），并在揭晓时记录 isBottomRevealed。

### 6.2 后续可选（非MVP）
- 用户登录系统、按用户保存游戏记录、排行榜、用户自定义海龟汤、多人对战等。

## 7. AI 交互规则（产品约束） — 明确化与实现细则
1. 输入：玩家任意自然语言问题（前端可做轻度规范化）。  
2. 输出：严格三值枚举：`是`、`否`、`无关`。任何非三值响应视为不合格并触发重试/兜底。  
3. 无关判定：
   - 问题无法从当前汤底信息推导出肯定/否定结论时返回 `无关`。  
   - 对越权请求（如“直接告诉我真相”、“把原因全说了”）统一返回 `无关`，并由前端展示友好提示（例如“该问题无法直接回答，请继续提问”）。  
4. 对话上下文与一致性：
   - 后端维护 confirmedFacts[]（已确认事实集合）以降低矛盾输出。  
   - 若 AI 返回与历史 confirmedFacts 冲突，优先以历史为准并触发记录与告警（可人工审查或自动修正策略）。  
5. 重试与兜底：
   - 若 provider 返回非三值或格式异常，服务端最多重试 N 次（默认 N=2，可配置）。  
   - 重试后仍不合规，服务端兜底返回 `无关` 并记录 providerResponseRaw 与 retriesCount 供排障。  
6. 合规与安全：
   - 避免 AI 越权解释汤底或提供额外信息；对敏感/滥用输入做速率限制与拦截。

## 10. 建议的接口与数据结构（开发用草案）
> 注：字段可微调，但需满足“每局一份汤底 + 提问历史 + 三值反馈”的闭环，并包含诊断用元数据。

### 10.1 GET /api/games
返回游戏卡片信息：id、title、difficulty、type、preview。

示例响应字段：
- id, title, difficulty, type, previewText, coverImageUrl

### 10.2 POST /api/game-sessions
入参：gameId 或 难度/类型筛选条件。  
出参（示例）：
- sessionId  
- soupPrompt（汤面故事）  
- createdAt  
- maxQuestions（配置）  
- askedCount（初始 0）

### 10.3 POST /api/game-sessions/:sessionId/ask
入参：
- question（玩家问题）

后端行为：
- 将 question 发给 Provider Adapter 调用 AI。  
- 校验 provider 返回仅为三值（严格匹配 `是`/`否`/`无关`）。  
- 若不合规，重试最多 N 次；最终兜底返回 `无关` 并记录 providerResponseRaw 与 retriesCount。

出参（必含）：
- answer（枚举：`是` / `否` / `无关`）  
- askedCount（当前会话提出问题总数）  
- retriesCount（本次请求的重试次数）  
- providerResponseRaw（仅供后端/运维/审核，前端可选择不显示）  
- timestamp

### 10.4 GET /api/game-sessions/:sessionId/bottom
出参：
- fullBottom（完整汤底与解析）  
- finalConclusion（摘要结论）  
- confirmedFacts[]（关键事实列表）  
- reasoningTrail（基于对话历史的“问题 -> 答案”时间序列）  
- isBottomRevealed（布尔）

## 11. 非功能性需求

### 性能
- AI 响应时间目标：p95 < 5s。若 Provider 超时，使用兜底逻辑并计为错误埋点。

### 可靠性
- 当 AI 返回非三值：触发重试或兜底为 `无关`，并将原始响应记录到日志供排查。  
- 建议启用 Provider 多节点或备用 Provider 以提高可用性（长期规划）。

### 安全与滥用
- 对 /ask 做速率限制（例如每会话最大频率、每 IP 限制）。  
- 日志记录必要信息用于排障，但默认脱敏或不记录敏感输入（providerResponseRaw 可脱敏或仅采样记录）。

## 12. 监控与埋点（必采集）
必采事件与字段（示例）：
- event: question_submitted -> {sessionId, questionLength, timestamp}  
- event: answer_returned -> {sessionId, answer, retriesCount, providerResponseValid(boolean), latencyMs}  
- event: session_created -> {sessionId, gameId, createdAt, maxQuestions}  
- event: bottom_revealed -> {sessionId, revealedAt}  
- 统计指标：提问次数分布、三值分布（是/否/无关）、重试率、provider 非合规率、平均/分位延迟（p50/p95）、错误率

告警示例：
- 三值合规率 < 99%  
- p95 > 5s  
- provider 非合规率激增或重试率异常

## 13. 测试与验收（新增）
自动化与手工测试用例应覆盖：

- 三值合规测试：随机问题集合，返回仅为三值。  
- 重试/兜底流程测试：模拟 provider 返回非三值、超时等，验证后端重试与兜底逻辑。  
- 一致性测试：长会话与 confirmedFacts 验证，确保不会前后矛盾。  
- 性能测试：确保 p95 < 5s。  
- 安全测试：速率限制与滥用场景。

验收标准（MVP + 新增项）：
- 游戏大厅能加载并开始游戏。  
- 游戏页面展示汤面、对话历史与问答功能。  
- /ask 接口对每次提问严格返回 `是`/`否`/`无关`（含重试/兜底）。  
- 汤底页展示完整真相与推理轨迹。  
- 埋点与监控能统计三值分布与响应时延。  
- 频率限制与日志记录生效。  
- 三值合规率 >= 99%（测验期可调整目标值）。

## 15. AI 提供方架构建议（新增）
实现 Provider Adapter 层以统一管理多个 AI 提供方：
- 统一接口：sendQuestion(sessionId, question) -> {answer, rawResponse, latency}

- 可配置策略：重试次数、超时、备用 Provider 顺序。  
- 支持 A/B 测试和治理（切换 DeepSeek / Claude）。

## 19. 附录：示例错误码与前端友好提示（建议）
- 200 OK -> 正常返回三值。  
- 429 TOO_MANY_REQUESTS -> 速率限制触发（前端提示“请求过于频繁，请稍后再试”）。  
- 503 PROVIDER_UNAVAILABLE -> Provider 超时或不可用（后端兜底返回 `无关`，前端可提示“AI 服务暂时不可用，已返回兜底答案”）。  
- 422 INVALID_RESPONSE -> Provider 返回内容不合规并已重试/兜底（记录 providerResponseRaw，前端显示通用反馈）。
